你是一名资深 B 端 / 政务 SaaS 视觉设计师。请为一套名为「盘古」的社区治理 / 业主自治后台设计 **PC Web 端管理框架**。

不要输出任何代码或技术实现，只输出：信息架构、框架布局、页面分区、组件选型、栅格与排版、配色与字阶。

## 1. 产品背景（决定信息架构）

盘古是面向「社区治理 / 业主自治」的多租户 SaaS。一个租户 = 一个小区。系统有三个使用端，**本次只设计 B 端 + G 端的 PC 后台**（C 端业主小程序不在范围内，但设计时要知道它的存在，因为后台的很多操作会"穿透"到业主端公示）：

- **B 端 · 业委会工作台**：业委会主任 / 委员使用。发起议题、核销资金、组织选举、管理委员会。
- **B 端 · 物业管理系统**：物业经理 / 物业客服使用。录入公共收益、提交开支单、处理维修工单。
- **G 端 · 街道办监管端**：街道办管理员 / 社区党组织书记使用。监管辖区多个小区、大额资金前置审查、换届纠纷处置。

**关键设计前提：这三类人用的是同一套后台外壳**，靠登录角色动态显隐左侧菜单和数据范围，不是三套独立系统。所以你要设计的是「一套角色自适应的外壳 + 一份菜单可见性矩阵」。

### 1.1 角色与数据范围（影响菜单显隐与页面顶部的"范围切换器"）

| 角色 | 端 | 数据范围 | 典型可见菜单 |
|---|---|---|---|
| 街道办管理员 | G | 辖区全部小区 | 全部模块 + 多小区切换 + 大额审查 + 强制撤销 |
| 社区党组织书记 | G | 单小区全量 | 大额资金前置审查、换届熔断处置、监管看板 |
| 业委会主任 | B | 本小区全量 | 议题、核销审批（密码B签名）、选举、委员会、报告 |
| 业委会委员 | B | 本小区全量（部分只读） | 议题、报告、沟通 |
| 楼栋代表 / 网格员 | B | 仅责任田楼栋（OWNER_GROUP） | 仅本楼栋议题催票、线下核销、维修工单 |
| 物业经理 | B | 仅本物业组织（ORG_ONLY） | 公共收益录入、开支单提交（密码A签名）、维修工单 |
| 物业客服 | B | 仅本物业组织 | 报修受理、工单处理 |
| 第三方审计师 | B | 本小区财务只读（按需激活） | 仅财务监督 + 内账导出 |

> 设计提示：左侧菜单要表现出"同一棵菜单树、不同角色看到不同子集"。请在交付物里附一张**菜单可见性矩阵图**，用「实心 = 可见可操作 / 半透明 = 只读 / 空 = 隐藏」三态表达。

## 2. 视觉风格基调（决定配色与字阶）

整体走 **「政务严肃蓝 + 数据看板科技感」**：底层稳重可信、符合政府监管调性；看板/数据区用渐变与图表点亮科技感。

### 2.1 配色 token（请据此建立 Color Styles）

- **主色 Primary**：政务蓝 `#1B4F9C`（按钮、选中态、关键强调）
- **主色深 / 浅**：`#143C78` / `#3A6FBF`
- **科技点缀 Accent**：青蓝 `#19A0C4` ~ 渐变到 `#2DD4BF`（仅用于看板数值、图表、进度条）
- **功能色**：成功 `#2E9E5B`、警告 `#E0A310`、危险 `#D14343`、信息 `#3A6FBF`
- **中性色**：背景 `#F4F6FA`、卡片 `#FFFFFF`、分隔线 `#E3E8F0`、正文 `#1F2733`、次要文字 `#5A6677`、占位 `#9AA5B5`
- **G 端监管区**：可在顶栏叠一条更深的靛蓝压条 `#0F2C5C`，强化"监管/权威"气质，与 B 端工作台做轻微区分（但仍是同一外壳）。
- **换届熔断 / 司法存证**：危险红 + 警示黄斜纹，用于熔断态横幅与锁定遮罩。

### 2.2 字阶（建立 Text Styles，中文优先）

字体：中文 `思源黑体 / PingFang SC`，数字与代码 `Inter / Roboto Mono`（哈希、金额、票数用等宽数字）。

- **栅格**：12 列，外边距 24，列间距 24，内容区最大宽 1440，超宽居中。
- **间距 token**：4 / 8 / 12 / 16 / 24 / 32 / 48（统一用 8 倍数体系，Auto Layout 间距取这些值）。
- **圆角**：卡片 8、按钮 6、Chip 4、模态 12。
- **阴影**：卡片用极轻投影（y2 blur8 黑 6%），悬浮态加重。
- **组件库（请建成可复用 Component）**：顶栏、侧栏菜单项（含三态）、面包屑、主/次/危险按钮、输入框、下拉、日期范围、数据范围切换器、统计卡(KPI Card)、数据表格(含批量操作栏/分页)、状态 Chip、进度环、时间轴/状态机步骤条、空状态、确认弹窗、抽屉(Drawer)、双签确认模态、文件/合同预览卡、哈希回执卡。

## 4. 信息架构全景（8 大模块导航骨架）

左侧菜单按以下 8 组组织（角色不同显隐不同，见 §1.1）。请把这 8 组全部画进导航骨架，标注每组下的二级页面。

1. **工作台 / 概览** — 角色自适应首页看板：待办、关键指标、换届熔断提醒、最新公示。
2. **用户与权限管理** — 业主名册、楼栋/单元结构、角色与数据范围配置、实名认证等级(L1/L3/L4)。
3. **物业管理** — 公共收益录入、物业费开支单（酬金制）、信托资金动用、物业组织信息。*（菜单内容随包干/酬金/信托三种模式变化，见 §6.4）*
4. **委员会操作** — 委员会名册、换届管理、会议纪要、职责分工。
5. **选举管理** — 候选人申报、资格审查、选举立项、选举投票看板、当选公示。*（物业角色此组整组隐藏）*
6. **议题与表决**（治理核心） — 议题立项、公示、表决看板、撤回、结算结果。
7. **财务监督** — 公共收益分配公示、资金流水、第三方审计导出、司法链存证记录。
8. **资产与维修** — 资产台账、维修工单（局部 vs 公共，ABAC 范围差异）、工程方案与验收。
9. *(辅助)* **沟通与报告** — 通知公告、定向推送记录、统计报表导出。

> 三种物业治理模式（包干制 / 酬金制 / 信托制）不是三个菜单，而是**同一组菜单在不同模式下的字段与流程差异**，详见 §6.4。请在「物业管理」与「财务监督」页面上用一个模式标识 Chip（包干/酬金/信托）表达当前小区所处模式。

外壳必须表现的 4 个特色元件：

1. **小区 / 辖区切换器**（顶栏左）：G 端街道办可在多个小区间切换并有"辖区汇总"选项；B 端固定单小区，显示为不可切换的小区名。
2. **物业模式标识 Chip**（顶栏）：显示当前小区是「包干制 / 酬金制 / 信托制」，不同颜色，hover 显示模式说明。
3. **换届熔断全局提醒**（顶栏右，常驻）：当小区进入 `HANDOVER_LOCK` 换届熔断态时，顶栏出现红色脉冲横幅"⚠ 本小区处于换届熔断期，大额资金划拨已锁定"，点击进入处置页。平时隐藏。
4. **数据范围标识**（内容区右上）：显示并（对 G 端）允许切换当前数据范围（辖区全部 / 本小区 / 本楼栋 / 本物业组织），呼应 ABAC。

请为外壳输出：①默认态（业委会主任视角）②G 端街道办视角（含辖区切换 + 监管压条）③换届熔断激活态 三个变体。

## 6. 七个重点业务页面（像素级深挖）

每个页面请给出：**业务目标 → 页面分区 → 关键组件 → 布局栅格 → 排版与信息层级**。

### 6.1 重点页 ①：议题表决看板（治理核心）

**业务目标**：管理员发起一个表决议题（如"小区主干道翻修 15 万元"），实时查看表决进度、参与情况、是否达到法定通过线。这套系统的表决遵循"双过半"红线（参与专有面积 ≥2/3 且 人数 ≥2/3），且分母随议题范围动态变化。

页面分区（建议上下两段 + 右侧信息栏）：
- **顶部议题信息条**：标题、类型 Chip（重大决议 / 一般决议 / 选举）、状态步骤条（草稿→公示→投票中→已截止→已结算，撤回为旁支终态）、起止时间倒计时。
- **核心进度看板（左 2/3）**：两个并排的大进度环——「专有面积通过率」与「人数通过率」，环中心是大号等宽数字百分比，环上标 2/3 红线刻度。下方一条"双过半达成"判定横幅（达成=绿、未达成=灰）。
- **动态分母说明卡（右 1/3）**：明确写出本议题的"投票分母从哪里来"。
  - 若是**公共维修 / 全局议题**：分母 = 全小区总面积 + 总户数（标"全局表决"）。
  - 若是**局部维修 / 单栋议题**（如 building_id=1001 1号楼屋面漏水）：分母 = 仅该楼栋面积 + 户数（标"局部共有·局部表决·局部分摊"），并明显标注"仅 1 号楼业主参与，其他楼栋无关"。
- **千人千面推送范围卡**：可视化本议题推送给了哪些人（全网 / 仅某楼栋），以及哪些楼栋代表/网格员的"催票与线下核销"工作台已解锁。
- **投票明细表（底部）**：业主投票流水（已投/未投、选项、是否 L3 刷脸认证）。注意：**投票进行中不暴露具体票数倾向给业主端，但管理端看板可见**——请在设计上区分"管理端可见票数 vs 业主端只见已投/未投"。

排版要点：进度环是视觉焦点，用科技青蓝渐变；红线刻度用危险红；分母说明用信息蓝卡片承载，文字分主/次两级讲清法理范围。

### 6.2 重点页 ②：信托制双签核销台

**业务目标**：信托制小区动用信托资金（如支付保安公司月度劳务费）必须"双密码双签"：物业经理输密码 A 签名提交 → 业委会主任收到强提醒 → 核对后输密码 B 签名 → 双签合并后通过国密签名打包写入最高院司法链，生成全网唯一交易哈希 TxHash。

页面分区：
- **左侧：待签 / 已签流水列表**，每条含金额、用途、发起物业、当前签名进度（A✓ B○）、状态 Chip。
- **右侧：选中单据的双签详情**，分三段竖向时间轴：
  1. **第一签（密码 A · 物业经理）**：申请人、金额、用途、附件发票、签名时间、密码 A 已签 ✓。
  2. **第二签（密码 B · 业委会主任）**：待当前用户操作时，显示醒目的 **"输入动态安全口令完成第二签"** 行动区（一个安全的密码输入 + 大号危险/主色确认按钮 + 强提醒"此操作将动用信托资金并永久上链，不可撤销"）。
  3. **司法链存证（自动）**：双签合并后展示 **哈希回执卡**——大号等宽 TxHash、"已写入最高院司法链"绿色徽标、链上时间戳、可点击"查看存证详情"。
- **强提醒机制可视化**：当有单据等待主任第二签时，顶栏消息区与本页都要有红点强提醒。

排版要点：双签时间轴是叙事主线，A 段已完成用低饱和、B 段待操作用主色高亮聚焦、链上回执用科技青蓝 + 等宽哈希字体强调"硬核存证"的技术质感。

### 6.3 重点页 ③：酬金制开支审批（待业委会初审状态机）

**业务目标**：酬金制小区，物业经理录入每一笔物业费开支（如买消防水带、修路灯），不立即扣款，自动进入"待业委会初审"状态机；业委会主任点【同意】后资金科目才核销。同时系统可一键激活第三方审计师全小区只读权限做离任审计。

页面分区：
- **开支单列表**：科目、金额、用途、提交物业、状态（待初审 / 已核销 / 已驳回）状态机 Chip。
- **审批详情抽屉 / 详情区**：开支明细、发票附件预览、状态机步骤条（物业提交 → 待业委会初审 → 已核销 / 已驳回）、主任的【同意核销】【驳回】操作区（驳回需填理由）。
- **第三方审计师入口卡**：一个显著的"激活年度酬金制离任审计"卡片——激活后审计师获得全小区财务纯只读权限，并可"一键导出全部物业费代理内账"。用一个权限激活开关 + 导出按钮表达。

排版要点：状态机步骤条横向贯穿详情顶部；金额用等宽数字右对齐；审计导出区用信息蓝边框卡与日常开支区分。

### 6.4 重点页 ④：物业管理 · 三模式差异页（公共收益 / 开支录入）

**业务目标**：同一个"物业管理"页，在三种治理模式下字段与可见范围完全不同。请用**同一页面框架 + 模式切换 = 三种内容变体**来表达，让视觉 AI 理解"模式驱动的差异化"。

- **包干制**：数据严格锁死"仅本组织"。后台**隐藏**"物业费内部行政开支""物业外包合同明细"等商业机密菜单，物业只录入代收的属于业委会的**公共收益**（车位费、电梯广告等）。页面只显示"公共收益录入"区，物业费内账区打灰/隐藏，并有说明"保障物业企业商业隐私与自主经营权"。
- **酬金制**：数据范围 ALL_COMMUNITY 只读，物业经理被次级扩权，可发起物业费开支单（连到 §6.3 状态机）。页面显示"公共收益 + 物业费开支单"两区，且顶部标"全小区只读 + 委员会审核"。
- **信托制**：物业降级为"职业经理人"，关联信托公共共有基金账户。页面显示"信托资金动用申请"入口（连到 §6.2 双签台），强调每笔支出将实时穿透到业主端 C 端看板。

排版要点：页面顶部一个"当前模式"大 Chip + 三模式说明 Tooltip；不同模式下用"隐藏/灰显"明确表达数据范围红线，而不是把所有字段都画出来。

### 6.5 重点页 ⑤：G 端社区党组织书记 · 大额资金前置审查页

**业务目标**：大额资金划拨（如主干道翻修 15 万）表决通过后，资金**不直接核销**，状态机切入"待社区党组织书记审核"。书记在 G 端调取招投标合同、预算明细做政治与合规前置审查（防突击花钱、防利益输送）。若审查期间触发换届纠纷，状态机变 `HANDOVER_LOCK` 熔断，划拨接口被死锁。

页面分区：
- **待审查工程列表**（左）：工程名、金额、所属小区、提交时间、状态。
- **审查详情（右）**：
  - 工程概览 + 大号金额。
  - **合规材料调阅区**：招投标合同、预算明细的文件预览卡（可逐份查看）。
  - **审查操作区**：【审查通过 · 放行划拨】（主色，需二次确认）/【驳回】（填理由）。
  - **换届熔断态遮罩**：若该小区处于换届熔断，整个放行操作区被**红色斜纹锁定遮罩**覆盖，文案"⚠ 换届熔断期，大额资金划拨接口已死锁，禁止放行"，按钮置灰不可点。
- **放行后**：显示"资金已通过 Outbox 划拨 + 支出已穿透上链，实时公示于业主端财务看板"的结果态。

排版要点：这是 G 端权威页，用更深的靛蓝顶压条；锁定遮罩是本页最强的视觉警示，必须让人一眼看出"被熔断锁死"。

### 6.6 重点页 ⑥：选举管理看板（换届）

**业务目标**：组织业委会换届选举——候选人申报、资格审查、选举立项、选举投票、当选公示。**此模块对物业角色整组隐藏**（物业无权介入业主自治选举）。

页面分区：
- **换届进度总览**：步骤条（申报 → 资格审查 → 立项 → 投票 → 计票 → 公示）。
- **候选人名册**：候选人卡片网格（姓名、楼栋、简介、资格审查状态 Chip）。
- **资格审查区**：逐个候选人的资格审查通过/驳回操作（带审查依据）。
- **选举投票看板**：复用 §6.1 的双过半进度环逻辑，但维度是选举得票。
- **当选公示**：当选名单 + 公示期倒计时。

排版要点：候选人用卡片网格而非表格，更具"人"的识别度；资格审查状态用三态 Chip。注意标注"此页对物业角色隐藏"。

### 6.7 重点页 ⑦：财务监督 / 公共收益公示（含司法链存证记录）

**业务目标**：财务透明是治理底线。本页给 B/G 端看资金流水、公共收益分配、第三方审计导出、以及所有上链存证记录。要表达"包干制只公示季度/年度公共收益分配，信托制则逐笔实时穿透"的差异。

页面分区：
- **顶部 KPI 卡组**：公共收益总额、本期支出、结余、待审支出（等宽大数字 + 同比小趋势）。
- **资金流水表**：逐笔收支（日期、科目、金额、对手方、模式相关的可见性）。信托制下每笔带"上链 ✓ TxHash"列。
- **公共收益分配公示卡**：包干制显示季度/年度分配公示；信托制显示"逐笔发票切片 + 实时穿透"说明。
- **司法链存证记录**：上链交易列表，每条一个哈希回执（TxHash、金额、时间、链上状态徽标）。
- **第三方审计导出区**：审计师只读视角下的"一键导出全部内账"。

排版要点：金额与哈希全用等宽数字；上链状态用科技青蓝徽标；KPI 卡是页面视觉锚点。

## 7. 其余四大模块（同样深挖）

同样按 **业务目标 → 页面分区 → 关键组件 → 布局栅格 → 排版与信息层级** 展开。这四组模块虽不如 §6 那样"治理特色"强烈，但承载了系统的底座数据与运营，需要完整设计。

### 7.1 用户与权限管理

这是整个 ABAC + RBAC 体系的"配置源头"，页面要让人理解"谁、在什么范围、能做什么"。

#### 7.1.1 业主名册

**业务目标**：维护小区全体业主档案，承载"一户多宅合并""开发商存量房""实名认证等级"等治理特有概念。投票时一户多宅按同一业主(uid)合并计票，所以名册要能表达这种归并关系。

页面分区：
- **顶部筛选栏**：楼栋/单元级联筛选、认证等级筛选（L1/L3/L4）、业主类型筛选（普通业主 / 开发商存量 / 委员）、关键词搜索。
- **业主数据表**：列含——业主姓名、联系方式（SM4 加密，默认脱敏显示 `138****6789`，有权限者可点击查看）、名下房产数（"一户多宅"用角标，如 `3 套`，hover 展开各套房号/面积）、专有面积合计、实名认证等级 Chip、业主类型 Chip、账户状态。
- **批量操作栏**：批量导出、批量发认证提醒。
- **业主详情抽屉**：基本信息、名下全部房产列表（房号、楼栋、面积、产权性质）、认证记录、投票委托关系（是否设为投票代理人 voting_delegate）。

排版要点：脱敏字段用占位灰 + 小锁图标表达"加密存储"；一户多宅角标用主色 Chip；面积/数量等宽右对齐。

#### 7.1.2 楼栋 / 单元结构（物理拓扑树）

**业务目标**：维护小区→楼栋→单元→房产的物理树。这棵树是 ABAC 数据范围（如 `CUSTOM_BUILDING:[1001]`）和"局部表决分母"的物理基础。

页面分区：**左侧树形**（小区 / 楼栋 / 单元层级，每节点显示户数、总面积）+ **右侧选中节点详情**（该楼栋基础信息、户数、总专有面积、关联的楼栋代表/网格员、该楼栋历史议题数）。支持节点增删改。

排版要点：树节点带统计角标（户数、面积）；选中态用主色高亮整行；详情区用 KPI 小卡呈现该节点的面积/户数（因为这就是计票分母来源）。

#### 7.1.3 角色与数据范围配置（RBAC + ABAC 核心配置页）

**业务目标**：给系统角色绑定权限点 + 数据范围。这是最能体现系统"权限即治理"特性的配置页：每个权限点有"法律红线"标记（红线权限不可随意授予），数据范围有四种类型。

页面分区（三栏）：
- **左：角色列表**（街道办管理员、党组织书记、业委会主任、委员、楼栋代表、网格员、物业经理、物业客服、第三方审计师…），选中查看其配置。
- **中：权限点穿梭/勾选区**，按 §4 的 8 大模块分组折叠；**法律红线权限**（如"强制撤销议题""大额放行""信托双签"）用红色盾牌图标标记并单独成组，提示"红线权限授予需谨慎"。
- **右：数据范围设置**，单选——
  - `辖区全部小区`（街道办）
  - `本小区全量`（主任/委员）
  - `仅责任田楼栋`（楼栋代表/网格员，带楼栋多选器，对应 `CUSTOM_BUILDING`）
  - `仅本物业组织`（物业，对应 `ORG_ONLY`）
  - `仅本人`（`SELF`）

排版要点：红线权限是本页视觉重点，用红盾 + 浅红底分组；数据范围用带说明的单选卡片，让配置者一眼看懂"这个范围意味着能看到多少数据"。

#### 7.1.4 实名认证等级管理

**业务目标**：业主认证分 L1（基础）/ L3（人脸核身）/ L4（更高级）。重大决议表决要求 L3 人脸认证才能投票，本页管理认证状态与升级。

页面分区：认证状态统计 KPI（各等级人数 + 未认证占比）、业主认证列表（当前等级、认证时间、是否满足重大决议门槛）、单个业主的认证记录时间轴。

排版要点：等级用三档颜色递进 Chip（L1 灰 / L3 主色 / L4 金）；用一个"重大决议可投票率"KPI 强调认证与表决资格的关联。

### 7.2 委员会操作

围绕业委会这个自治组织的"人"与"届"运转，并与换届熔断强相关。

#### 7.2.1 委员会名册

**业务目标**：维护当届业委会成员（主任、副主任、委员）名单、职务、联系方式、任期。

页面分区：成员卡片网格（头像、姓名、职务 Chip、所属楼栋、任期起止）+ 当届概览条（届次、成立日期、任期剩余、成员数）。

排版要点：用卡片网格突出"人"；主任卡片用主色描边强调核心角色。

#### 7.2.2 换届管理（与 HANDOVER_LOCK 强相关）

**业务目标**：管理业委会换届全流程，并在换届纠纷期触发熔断态——这是 §6.5 大额资金锁定的源头。换届期间的状态要在全系统可感知。

页面分区：
- **换届进度步骤条**：换届公告 → 选举（跳转 §6.6）→ 新老交接 → 完成。
- **届次对照卡**：上一届 vs 新一届成员对照。
- **换届熔断控制区**（G 端 / 书记可见）：当出现换届纠纷时，可将小区置为 `HANDOVER_LOCK` 熔断态——一个醒目的红色"熔断开关"卡片，开启后全局顶栏横幅亮起、大额资金接口锁定。卡片要写明"熔断将冻结大额资金划拨，防止离任突击花钱"。
- **交接清单**：账册、印章、资料的交接核对清单。

排版要点：熔断控制区是本页最强警示元件，红色斜纹 + 二次确认；非熔断态时该区显示为"当前无熔断，运行正常"的安全绿态。

#### 7.2.3 会议纪要 & 7.2.4 职责分工

**业务目标**：归档业委会会议纪要（议题、决议、出席）、维护成员职责分工矩阵。

页面分区：会议纪要 = 列表（日期、议题、决议摘要、附件）+ 详情（富文本纪要 + 出席名单 + 关联议题链接）；职责分工 = 成员 × 职责的矩阵表。

排版要点：纪要详情用文档式留白排版；职责矩阵用清晰的二维表 + 勾选标记。

### 7.3 资产与维修工单

承载"局部 vs 公共"维修的非对称业务流，与 §6.1 的动态分母、换届熔断一脉相承。

#### 7.3.1 资产台账

**业务目标**：维护小区公共资产（电梯、监控、消防、路面、绿化等）台账，是维修工单的关联对象。

页面分区：筛选栏（资产类别、所属楼栋/公共区域、状态）+ 资产数据表（资产名、类别、位置、关联物理节点 building_id 或"全局公共"、状态、最近维修）+ 资产详情（基础信息 + 维修历史时间轴）。

排版要点：用"所属范围"Chip 区分**楼栋专属资产**（如 1 号楼电梯）与**全局公共资产**（如主干道、中央监控）——这直接决定后续维修走局部还是公共流程。

#### 7.3.2 维修工单（局部 vs 公共，ABAC 范围非对称）

**业务目标**：这是资产模块的核心，要清晰表达两条**非对称异构**的业务流：

- **局部维修**（如 1 号楼顶层漏水 / 某单元电梯故障）：工单关联特定 `building_id`，原则"局部共有·局部表决·局部分摊"。表决分母只取该楼栋，资金走该楼栋专项维修资金，**不波及其他楼栋**。仅该楼栋业主与楼栋代表参与。
- **公共维修**（如主干道破损 / 中央监控瘫痪）：全局共有，全局表决，动用公共收益或全局维修资金，全网推送，所有网格员/楼栋代表工作台联动。大额时进入 §6.5 书记前置审查。

页面分区：
- **工单列表**：工单号、标题、**范围 Chip（🏠局部·楼栋号 / 🌐公共·全局）**、关联资产、资金来源、状态机 Chip、金额。范围 Chip 用不同颜色强区分。
- **工单详情**：
  - 报修信息（现场照片、报修人、关联物理节点）。
  - **状态机步骤条**：报修受理 → 勘察出方案 → 表决（局部/全局，跳 §6.1）→ 施工 → 验收 → 资金核销 → 归档。若处于换届熔断，状态条上叠加"⚠ 熔断锁定"标记，资金核销节点被锁。
  - 工程方案卡（预算、施工方、资金来源专户）。
  - **ABAC 范围可视化**：用一张小拓扑图或文字明确"本工单影响范围 = 仅 1 号楼 / 全小区"，呼应分母与分摊。
- **新建工单表单**：选择范围（局部需选楼栋 / 公共为全局），系统据此决定后续表决分母与推送范围。

排版要点：局部 vs 公共的视觉区分是本页灵魂——用范围 Chip 颜色、影响范围说明、资金来源专户三处反复强化"局部不侵染其他楼栋"。

#### 7.3.3 工程方案与验收

**业务目标**：录入工程预算方案、上传招投标合同、施工完成后上传验收报告。

页面分区：方案信息（造价、工期、施工方、资金来源）+ 合同/预算附件预览卡 + 验收区（验收报告上传、验收结论、关联工单）。

排版要点：附件用可预览的文件卡（呼应 §6.5 书记调阅合同）；金额等宽。

### 7.4 沟通与报告

承载"千人千面定向推送"的运营出口与数据报表。

#### 7.4.1 通知公告

**业务目标**：发布面向业主/委员的通知公告，可设定推送范围。

页面分区：公告列表（标题、范围、发布时间、阅读率）+ 发布表单（标题、富文本正文、附件、**推送范围选择器**：全小区 / 指定楼栋 / 指定角色）+ 公告详情（含阅读统计）。

排版要点：推送范围选择器要直观（全局 / 楼栋多选 / 角色多选），与下面的定向推送同源。

#### 7.4.2 定向推送记录（千人千面可视化）

**业务目标**：系统通过消息队列做"千人千面"定向推送——如局部维修议案只推给该楼栋业主。本页回看每次推送的**精准触达画像**。

页面分区：
- **推送记录列表**：推送主题、关联议题/工单、推送范围（全网 / 某楼栋）、目标人数、送达率、时间。
- **单次推送画像详情**：用一张范围可视化图表达"这条消息推给了谁"——如"仅 1 号楼 86 户 / 全小区 1240 户"，并标注哪些楼栋代表/网格员工作台因此被解锁催票权限。

排版要点：触达画像是本页特色，用对比柱状/范围高亮表达"精准 vs 全网"；触达率用进度条 + 等宽百分比。

#### 7.4.3 统计报表导出

**业务目标**：导出治理运营报表（表决参与率、财务收支、工单时效、认证率等），供 G 端监管与年审使用。

页面分区：报表类型卡片网格（每类一卡：图标 + 名称 + 说明 + 导出按钮）+ 报表预览区（选中类型后展示图表预览：参与率趋势、收支结构、工单时效分布等）+ 导出参数（时间范围、小区范围、格式）。

排版要点：图表预览区用看板科技风（青蓝渐变图表）；报表卡网格整齐，导出动作明确。

## 9. 约束与红线（设计时务必遵守）

- **同一外壳、角色显隐**：不要画三套割裂的系统；用一套外壳 + 菜单三态表达角色差异。
- **数据范围红线要可见**：包干制隐藏商业机密菜单、网格员只见责任田楼栋——这些"看不到"本身就是设计的一部分，请用灰显/隐藏明确表达。
- **不可撤销操作要有强警示**：双签上链、大额放行、强制撤销议题，都要二次确认 + 后果文案。
- **换届熔断是全局态**：熔断时顶栏横幅 + 相关操作锁定遮罩，必须强烈可感知。
- **金额 / 票数 / 哈希用等宽数字**，右对齐，强调严谨。
- **业主端 vs 管理端信息差**：表决进行中，业主端只见"已投/未投"，管理端看板可见票数——设计上注明这一区别。
